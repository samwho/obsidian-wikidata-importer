# AGENTS.md

## Project overview

This is an Obsidian community plugin named **Wikidata Importer**. Its plugin ID is `wikidata-importer`, from `manifest.json`.

The plugin imports Wikidata item data into Obsidian note frontmatter. Users can:

- Add a Wikidata item ID, such as `Q42`, to a configured frontmatter key, then run `Wikidata Importer: Import properties for active file`.
- Search for a Wikidata entity and create or update a note with imported properties.
- Search from selected editor text with `Wikidata Importer: Import entity from highlighted text`.

The implementation uses Obsidian's `Plugin`, `SuggestModal`, `PluginSettingTab`, `Notice`, `TFile`, `MarkdownView`, `requestUrl`, and `fileManager.processFrontMatter` APIs.

## Important files

- `main.ts`: plugin entry point, Obsidian command registration, frontmatter sync, settings tab, and entity search modal.
- `src/wikidata.ts`: Wikidata API/SPARQL access and conversion of Wikidata results into Obsidian frontmatter values.
- `manifest.json`: Obsidian plugin metadata. The `id` should match the plugin folder name for local development.
- `versions.json`: maps plugin versions to minimum supported Obsidian app versions.
- `esbuild.config.mjs`: bundles `main.ts` into `main.js`; keeps Obsidian, Electron, CodeMirror, Lezer, and Node built-ins external.
- `styles.css`: plugin stylesheet included in release assets, currently only template comments.
- `.github/workflows/release.yml`: builds on pushed tags and creates a draft GitHub release.
- `version-bump.mjs`: npm version hook that copies `package.json`'s version into `manifest.json` and updates `versions.json`.

Generated/runtime files:

- `main.js` is generated by esbuild and should not be edited by hand.
- `data.json` is local Obsidian plugin settings and should not be committed.
- `node_modules/` is local dependency state.

## Development workflow

Install dependencies:

```bash
npm install
```

Start watch mode:

```bash
npm run dev
```

This runs `node esbuild.config.mjs`, watches TypeScript sources, and writes `main.js` in the plugin root. Because this repository is already inside an Obsidian vault plugin directory, Obsidian can load it directly from:

```text
.obsidian/plugins/obsidian-wikidata-importer/
```

After source changes, reload the plugin in Obsidian by disabling and enabling it, using Obsidian's "Reload app without saving" command, or using a hot-reload plugin. Restart Obsidian after changing `manifest.json`; Obsidian only reloads manifest metadata on app restart.

Production build:

```bash
npm run build
```

This runs TypeScript checking with `tsc -noEmit -skipLibCheck`, then bundles with esbuild in production mode. Production output omits inline sourcemaps.

There is currently no test script or lint script in `package.json`. For this repo, `npm run build` is the main verification command.

## Code notes

- Keep changes scoped to TypeScript sources unless release metadata or docs need updates.
- Do not edit generated `main.js` directly; regenerate it with `npm run dev` or `npm run build`.
- `syncEntityToFile` is the central frontmatter update path. It fetches Wikidata properties, applies allow/block settings, preserves existing properties unless overwrite is enabled, and always writes the configured entity ID key.
- `Entity.search` calls Wikidata's `wbsearchentities` API once per configured language.
- `Entity.getProperties` queries the Wikidata SPARQL endpoint and converts strings, dates, decimals, integers, and linked Wikidata items into frontmatter-compatible values.
- Internal links are built with the configured `internalLinkPrefix`; `${label}` and `${id}` placeholders are supported.
- Settings are persisted through Obsidian's `loadData`/`saveData`.

## Versioning

Use npm's version command so the repo's version metadata stays aligned:

```bash
npm version patch
npm version minor
npm version major
```

The `version` lifecycle script runs:

```bash
node version-bump.mjs && git add manifest.json versions.json
```

That updates:

- `package.json`
- `package-lock.json`
- `manifest.json`
- `versions.json`

Before releasing, make sure those files agree on the intended version. At the time this file was written, `package.json`, `package-lock.json`, and `manifest.json` were not all on the same version, so check this explicitly before tagging.

Only change `manifest.json`'s `minAppVersion` when the plugin starts depending on a newer Obsidian API. `versions.json` is used by Obsidian to find the newest plugin version compatible with older app versions.

## Release workflow

1. Verify the code:

   ```bash
   npm install
   npm run build
   ```

2. Bump the version with `npm version patch`, `npm version minor`, or `npm version major`.

3. Confirm `manifest.json`'s `version` matches the git tag exactly. Obsidian expects the release tag to match the manifest version, without a `v` prefix.

4. Push the commit and tag:

   ```bash
   git push
   git push --tags
   ```

5. The GitHub Actions release workflow runs on any pushed tag. It installs dependencies, builds the plugin, and creates a draft GitHub release containing:

   - `main.js`
   - `manifest.json`
   - `styles.css`

6. Review the draft release, add release notes, and publish it.

For manual releases, create a GitHub release whose tag exactly matches `manifest.json`'s `version`, then upload `main.js`, `manifest.json`, and optionally `styles.css` as binary release assets.
